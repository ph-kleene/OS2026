# OS 作业 3 答案

## 第 1 题：pthread 读写锁行为测试

题目要求：采用 C 语言实现测试 pthread 读写锁行为的多线程程序，利用 sleep 设定线程顺序和持续时间，测试不同设置下读写锁的行为特征（读优先/写优先/读写公平），用甘特图展示结果并分析。

### 1.1 设计思路

#### 1.1.1 pthread 读写锁的三种属性

Linux glibc 提供 `pthread_rwlockattr_setkind_np()` 设置读写锁行为：

| 属性常量 | 行为 |
|---------|------|
| `PTHREAD_RWLOCK_PREFER_READER_NP` | 读优先：当读锁被持有时，新到达的读者仍可获取锁（即使有写者在等待），可能导致**写者饥饿** |
| `PTHREAD_RWLOCK_PREFER_WRITER_NONRECURSIVE_NP` | 写优先：当写者在等待时，新到达的读者被阻塞，写者优先获取锁。避免写者饥饿 |
| `PTHREAD_RWLOCK_PREFER_WRITER_NP` | 写优先（另一种实现），但 glibc 2.35 中行为与 default 基本一致 |
| default（不设置属性） | 与 `PTHREAD_RWLOCK_PREFER_READER_NP` 行为相同 |

#### 1.1.2 测试方法

测试程序的核心设计：

- **精确计时**：使用 `clock_gettime(CLOCK_MONOTONIC)` 获取高精度时间戳
- **同步启动**：使用 `pthread_barrier` 确保所有线程同时开始执行
- **可控顺序**：通过 `nanosleep` 在请求锁之前插入不同的延迟，控制线程到达顺序
- **事件记录**：每个线程记录自己的"开始等待"、"获取锁"、"释放锁"三个时间点
- **甘特图输出**：根据记录的时间数据，用 ASCII 字符绘制甘特图

#### 1.1.3 测试场景设计

设计了 4 个核心场景，覆盖不同竞争模式：

| 场景 | 线程配置 | 测试目的 |
|------|---------|---------|
| A | R1(hold=3s) → W2(delay=0.5s) → R3(delay=1s) | 读者持锁时，新到达的读者是否能共享锁？写者何时获取锁？ |
| B | W1(hold=3s) → R2(delay=0.5s) → W3(delay=1s) → R4(delay=1.5s) | 写者释放锁后，等待的读者和写者谁优先？ |
| C | R1+R2(hold=2s), W3(delay=0.7s), R4(delay=1.2s) | 验证读优先模式下写者饥饿的场景 |
| D | W1(hold=2s), R2(delay=0.3s), W3(delay=0.6s), R4(delay=0.9s) | 多个写者和读者混合到达的排序行为 |

### 1.2 源码 `q1_src/rwlock_test.c`

#### 核心数据结构

```c
typedef enum { OP_READ, OP_WRITE } op_t;

typedef struct {
    int      tid;          /* thread id */
    op_t     op;           /* reader or writer */
    long     wait_start;   /* when the thread first tried to lock (ms) */
    long     hold_start;   /* when it actually acquired the lock (ms) */
    long     hold_end;     /* when it released the lock (ms) */
} event_t;
```

#### 线程函数

```c
static void *rw_thread(void *raw)
{
    thread_arg_t *a = (thread_arg_t *)raw;

    pthread_barrier_wait(a->barrier);  /* all threads start together */

    /* delay to control arrival order */
    struct timespec ts = { .tv_sec = a->delay_ms / 1000,
                           .tv_nsec = (a->delay_ms % 1000) * 1000000L };
    nanosleep(&ts, NULL);

    long ws = now_ms();

    if (a->op == OP_READ)
        pthread_rwlock_rdlock(a->rwlock);
    else
        pthread_rwlock_wrlock(a->rwlock);

    long hs = now_ms();

    /* hold the lock for the specified duration */
    ts.tv_sec  = a->hold_ms / 1000;
    ts.tv_nsec = (a->hold_ms % 1000) * 1000000L;
    nanosleep(&ts, NULL);

    pthread_rwlock_unlock(a->rwlock);
    long he = now_ms();

    record(a->tid, a->op, ws, hs, he);
    return NULL;
}
```

### 1.3 测试结果与甘特图

#### 场景 A：R1 持锁 → W2 到达 → R3 到达

这是最经典的读写锁行为测试。R1 作为读者持有锁 3 秒；W2（写者）在 0.5 秒后尝试获取锁；R3（读者）在 1 秒后尝试获取锁。

**关键问题**：R3 能否与 R1 共享锁（读优先），还是必须等待 W2 先获取（写优先）？

##### Default（读优先）

```
time (ms)  0       420     840     1260    ...
           ├───────├───────├───────├───────├── ...
T1  [R]  █████████████████████████████████████████████████████████  wait=0ms    hold=3000ms
T2  [W]           ················································  wait=2500ms hold=1000ms
T3  [R]                     ███████████████████                    wait=0ms    hold=1000ms

行为：R3 在 t=1s 到达，立即获取锁与 R1 共享。W2 必须等待所有读者离开。
```

**分析**：R3 的 wait=0ms，说明 R3 到达时立即获得了读锁（与 R1 共享）。W2 等待了 2500ms（从 t=0.5s 到 t=3s），直到 R1 释放锁后才能写入。这是典型的**读优先**行为——读者可以不断加入，写者被推迟。

##### PREFER_READER_NP

```
T1  [R]  █████████████████████████████████████████████████████████  wait=0ms    hold=3000ms
T2  [W]           ················································  wait=2500ms hold=1001ms
T3  [R]                     ███████████████████                    wait=0ms    hold=1000ms

行为：与 default 完全相同。R3 即时获取，W2 等待。
```

**分析**：`PREFER_READER_NP` 与默认行为一致。R3 获得即时访问，W2 被延迟到所有读者完成后。

##### PREFER_WRITER_NONRECURSIVE_NP（写优先）

```
T1  [R]  ██████████████████████████████████████████████            wait=0ms    hold=3001ms
T2  [W]         ·······································████████████ wait=2500ms hold=1000ms
T3  [R]                 ··········································  wait=3001ms hold=1000ms

行为：R1 释放后，W2 先获取锁。R3 必须等待 W2 完成。
```

**分析**：这里 R3 的 wait=3001ms，意味着 R3 虽然比 W2 到达晚，并且在 R1 释放前就已到达（t=1s < t=3s），但却不能与 R1 共享锁！R3 在 R1 释放后也不能立即获取锁——必须等 W2 先完成。这是**写优先**的关键特征：**当写者在等待时，即使有读者正在读取，新到达的读者也被阻止进入**，且写者优先于已在等待的读者获取锁。

##### PREFER_WRITER_NP

```
T1  [R]  █████████████████████████████████████████████████████████  wait=0ms    hold=3000ms
T2  [W]           ················································  wait=2497ms hold=1000ms
T3  [R]                     ███████████████████                    wait=0ms    hold=1000ms

行为：与 default 和 PREFER_READER_NP 相同。R3 即时获取。
```

**分析**：在 glibc 2.35 中，`PTHREAD_RWLOCK_PREFER_WRITER_NP` 的行为与读优先一致。这是因为 glibc 自 2.25 版本起改变了 rwlock 的默认实现，使其使用了更公平的算法。`PREFER_WRITER_NP` 在源码中实际上行为等同于 `PREFER_READER_NP`。真正实现写优先的是 `PREFER_WRITER_NONRECURSIVE_NP`。

#### 场景 B：W1 持锁 → R2、W3、R4 到达

W1 作为写者持有锁 3 秒；R2（读 delay=0.5s）、W3（写 delay=1s）、R4（读 delay=1.5s）陆续到达。

**关键问题**：W1 释放后，等待队列中的读者和写者谁先获取锁？

##### PREFER_READER_NP

```

T2  [R]         ·······································████████████ wait=2500ms hold=1001ms
T3  [W]                 ··········································  wait=3001ms hold=1000ms

行为：W1 释放后，R2 和 R4 同时获取读锁（共享）。W3 必须等读者全部完成。
```

**分析**：读优先策略下，W1 释放后，等待的读者（R2 和 R4）同时获得锁并共享读取。W3（写者）必须等待所有读者完成。注意 R4 的 wait=1501ms，说明它从到达（t=1.5s）到获取锁（t≈3s）等待了约 1.5s，但在读优先策略下，它不需要等待 W3（写者），因为读者可以"插队"。

##### PREFER_WRITER_NONRECURSIVE_NP

```
T1  [W]  ██████████████████████████████████████████████            wait=0ms    hold=3001ms
T2  [R]         ··················································  wait=3500ms hold=1000ms
T3  [W]                 ·······························████████████ wait=2001ms hold=1000ms
T4  [R]                         ··································  wait=2501ms hold=1000ms

行为：W1 释放后，W3 优先获取锁。R2 和 R4 必须等 W3 完成。
```

**分析**：写优先策略下，W1 释放后，虽然 R2 比 W3 更早开始等待，但 W3（写者）获得了优先权。R2 的 wait=3500ms（从 t=0.5s 到 t≈4s），这意味着 R2 必须等待两个写者（W1→W3）都完成后才能读取。R4 同样被推迟。这展示了写优先如何防止写者饥饿——**代价是降低了读并发性**。

### 1.4 分析结论

通过上述 4 个场景，在 4 种属性下的测试，可以得出以下结论：

1. **Linux glibc 2.35 默认行为是读优先**。`default`、`PREFER_READER_NP` 和 `PREFER_WRITER_NP` 三者行为相同——读者可以持续获取锁，写者可能饥饿。
2. **真正的写优先仅由 `PREFER_WRITER_NONRECURSIVE_NP` 实现**。该模式下，当写者在等待时，新读者被阻止，写者按 FIFO 顺序获取锁。
3. **读写锁的本质权衡**发生在**读并发性**和**写者饥饿**之间：
   - 读优先：最大化读吞吐量，但写者可能在读密集型负载中饥饿
   - 写优先：避免写者饥饿，但降低了读并发性
4. **"公平"是一个相对概念**：Linux pthread 的写优先模式更接近于"写者不被饿死"而非"严格 FIFO"。读者之间的并发关系在写优先模式下仍然存在（场景 B 中 R2 和 R4 在获取锁后仍可并发读取）。
5. **选择建议**：
   - 读多写少的场景 → 使用默认（读优先），最大化读性能
   - 写操作延迟敏感 → 使用 `PREFER_WRITER_NONRECURSIVE_NP`，避免写者饥饿
   - 注意 `PREFER_WRITER_NONRECURSIVE_NP` 不支持递归写锁，重入会导致死锁

## 第 2 题：环形缓冲区 ringBuffer

题目要求：采用 C++ 实现存放 int 的环形缓冲区类，提供 put/get 函数，结合 pthread 条件变量和互斥锁实现多线程同步，并用多线程测试程序验证正确性。

### 2.1 设计思路

#### 2.1.1 同步模型

采用**管程（Monitor）模型**实现 ringBuffer：

- 一个 `pthread_mutex_t` 保护所有共享状态（`head_`、`tail_`、`count_`、`buf_`）
- 两个条件变量：
  - `not_full_`：缓冲区有空位时通知等待的生产者
  - `not_empty_`：缓冲区有数据时通知等待的消费者

```
        生产者线程                  消费者线程
           │                          │
           ▼                          ▼
       mutex_lock                 mutex_lock
           │                          │
      while(full)                while(empty)
      cond_wait(not_full)        cond_wait(not_empty)
           │                          │
      buf[head]=val              val=buf[tail]
      head=(head+1)%cap          tail=(tail+1)%cap
      count++                    count--
           │                          │
    cond_signal(not_empty)     cond_signal(not_full)
           │                          │
       mutex_unlock               mutex_unlock
```

#### 2.1.2 环形缓冲区实现

使用三个指针维护状态：

- `head_`：下一个写入位置
- `tail_`：下一个读取位置
- `count_`：当前元素数量

当 `head_ == tail_` 时，如果 `count_ == 0` 则为空，`count_ == cap_` 则为满。使用 `count_` 区分两者，避免浪费一个槽位。

#### 2.1.3 为什么使用 `while` 而非 `if`

```cpp
while (count_ == cap_)
    pthread_cond_wait(&not_full_, &mutex_);
```

使用 `while` 循环而非 `if` 的原因：

1. **虚假唤醒（Spurious Wakeup）**：POSIX 允许 `pthread_cond_wait` 在未被 `signal`/`broadcast` 的情况下返回。
2. **多消费者竞争**：当多个消费者同时等待时，一个 `signal` 只唤醒一个，但被唤醒的线程可能在实际获取 mutex 之前又被其他线程抢先消耗了数据。`while` 确保重新检查条件。

这是一个经典的管程编程实践——**永远在循环中检查条件**（Mesa-style monitor semantics）。

### 2.2 源码

#### 2.2.1 头文件 `q2_src/ringBuffer.h`

```cpp
#ifndef RINGBUFFER_H
#define RINGBUFFER_H

#include <pthread.h>
#include <cstddef>

class ringBuffer {
public:
    explicit ringBuffer(size_t capacity);
    ~ringBuffer();

    /* copy / move prohibited — owns mutex/cv resources */
    ringBuffer(const ringBuffer &) = delete;
    ringBuffer &operator=(const ringBuffer &) = delete;

    void put(int value);          /* blocking: waits while full  */
    void get(int &out);           /* blocking: waits while empty */

    size_t size() const;
    size_t capacity() const;
    bool   empty() const;
    bool   full() const;

private:
    int            *buf_;
    size_t          cap_;
    size_t          head_;
    size_t          tail_;
    size_t          count_;

    pthread_mutex_t mutex_;
    pthread_cond_t  not_full_;
    pthread_cond_t  not_empty_;
};

#endif
```

#### 2.2.2 实现 `q2_src/ringBuffer.cpp`

核心的 put 和 get 方法：

```cpp
void ringBuffer::put(int value)
{
    pthread_mutex_lock(&mutex_);

    /* wait while the buffer is full */
    while (count_ == cap_)
        pthread_cond_wait(&not_full_, &mutex_);

    buf_[head_] = value;
    head_ = (head_ + 1) % cap_;
    count_++;

    /* signal one waiting consumer */
    pthread_cond_signal(&not_empty_);
    pthread_mutex_unlock(&mutex_);
}

void ringBuffer::get(int &out)
{
    pthread_mutex_lock(&mutex_);

    /* wait while the buffer is empty */
    while (count_ == 0)
        pthread_cond_wait(&not_empty_, &mutex_);

    out = buf_[tail_];
    tail_ = (tail_ + 1) % cap_;
    count_--;

    /* signal one waiting producer */
    pthread_cond_signal(&not_full_);
    pthread_mutex_unlock(&mutex_);
}
```

#### 2.2.3 测试程序 `q2_src/test_ringbuffer.cpp`

测试程序设计要点：

- **基本正确性测试**：单线程环境下验证 FIFO 顺序、empty/full 状态
- **多线程压力测试**：4 个生产者 + 4 个消费者，每个生产者 200,000 条数据，缓冲区容量仅 16（故意制造高竞争）
- **完整性验证**：每个生产者累计本地 sum，每个消费者累计本地 sum，最后比较 total_produced_sum == total_consumed_sum 以及 count 匹配。**使用 per-thread stats 避免共享 mutex 的额外竞争**
- **优雅终止**：生产者完成后，主线程向缓冲区注入 NUM_CONSUMERS 个 sentinel 值（-1），消费者收到 sentinel 后退出。这种 "poison pill" 模式避免了复杂的共享标志和额外的同步

### 2.5 设计思考与改进点

#### 2.5.1 `signal` vs `broadcast`

在 put/get 中使用 `pthread_cond_signal` 而非 `pthread_cond_broadcast`：

- put 后只需要唤醒**一个**等待的消费者（因为只放入了一个元素）
- get 后只需要唤醒**一个**等待的生产者（因为只空出了一个槽位）
- broadcast 会唤醒所有等待者，导致"惊群效应"（thundering herd），大量线程竞争 mutex 后又发现条件不满足继续睡眠，浪费 CPU

#### 2.5.2 不可复制性

```cpp
ringBuffer(const ringBuffer &) = delete;
ringBuffer &operator=(const ringBuffer &) = delete;
```

ringBuffer 持有 pthread 同步原语（mutex、condvar）的所有权和原始指针（buf_），复制会导致双重释放、死锁等问题。显式删除复制构造和赋值运算符是必要的防御性设计。

#### 2.5.3 性能瓶颈分析

当前实现的性能瓶颈在于每个 put/get 都需要获取和释放 mutex。在容量为 16、8 个线程竞争的场景下，2.4 秒完成 80 万次操作——平均每次操作约 3μs。这个性能对于教学目的已经很好，但在极致性能场景下可以考虑：

- **无锁（lock-free）环形缓冲区**：使用原子操作替代 mutex，但实现复杂度显著增加
- **批处理**：一次获取 mutex 后处理多个元素，减少锁开销
- **双缓存/多缓存**：多个缓冲区轮换，减少竞争

#### 2.5.4 与课程参考代码的对比

课程提供的 `ringBuffer.cpp` 使用信号量（semaphore）实现同步。本实现改用条件变量（condition variable）配合互斥锁，这是管程的经典模式。两者的区别：

| 维度 | 信号量方案 | 条件变量方案 |
|------|----------|------------|
| 语义表达 | 计数信号量，需要小心维护计数 | `while(条件) wait()` 直接表达等待条件 |
| 易错性 | 信号量的 P/V 操作容易放错位置 | 条件变量有明确的 mutex 保护，更不易出错 |
| 可扩展性 | 增加条件判断需要额外信号量 | 新增条件只需增加条件变量 |

条件变量方案更符合结构化并发的思想——**在互斥保护下等待特定条件**——这也是现代并发编程的主流范式。
