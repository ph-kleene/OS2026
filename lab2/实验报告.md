## 三、代码实现说明

### 1. 内核初始化

- [workspace/OS/lab2/kernel/x86/idt.c](workspace/OS/lab2/kernel/x86/idt.c)
  - 设置 0x80 系统调用门，DPL=3，允许用户态触发 int 0x80。
  - 设置 0x20 时钟中断门，DPL=0，仅内核态可用。

- [workspace/OS/lab2/kernel/device/timer.asm](workspace/OS/lab2/kernel/device/timer.asm)
  - 初始化 8253 PIT。
  - 控制字写入 0x43，分频值 11932 写入 0x40（低字节先写，高字节后写）。
  - 实现 100Hz 中断频率（10ms 一次）。

- [workspace/OS/lab2/kernel/x86/gdt.c](workspace/OS/lab2/kernel/x86/gdt.c)
  - 在 enterUserSpace 中调用 switch_to_user，进入 Ring 3 执行用户程序。

### 2. 中断入口与系统调用内核处理

- [workspace/OS/lab2/kernel/x86/trap_entry.asm](workspace/OS/lab2/kernel/x86/trap_entry.asm)
  - 实现 irqSyscall，压入 err_code=0 与 irq=0x80 后进入公共中断入口。
  - 实现 asmDoIrq：保存现场、切换内核段寄存器、调用 irqHandle、恢复现场并 iret 返回。

- [workspace/OS/lab2/kernel/core/irq_dispatch.c](workspace/OS/lab2/kernel/core/irq_dispatch.c)
  - syscallHandle 按 eax 分发：SYS_WRITE、SYS_NOW、SYS_ZERO_TIME_TICKS、SYS_GET_TIME_TICKS。
  - syscallWrite 将用户态缓冲区地址按段基址转换后写入 VGA，处理换行/回车与滚屏。
  - syscallNow 从 RTC 读取 BCD 编码并转换成十进制时分秒。
  - timerHandle 递增 ticks 并向主片 PIC 发送 EOI。

### 3. 用户态库封装

- [workspace/OS/lab2/lib/doSyscall.asm](workspace/OS/lab2/lib/doSyscall.asm)
  - 按调用约定装载 eax/ecx/edx/ebx/esi/edi，触发 int 0x80，返回值在 eax。

- [workspace/OS/lab2/lib/syscall.c](workspace/OS/lab2/lib/syscall.c)
  - now：封装 SYS_NOW。
  - sleep：基于 ticks（100Hz）忙等延时。
  - printf：实现基本格式输出（%d、%s、%c、%%，支持宽度如 %2d），通过 SYS_WRITE 输出。

### 3.1 运行稳定性修复

- [workspace/OS/lab2/kernel/x86/i8259.asm](workspace/OS/lab2/kernel/x86/i8259.asm)
  - 初始化 PIC 后显式设置中断屏蔽位：主片仅放行 IRQ0（定时器），从片全部屏蔽。
  - 目的：确保时钟中断稳定到达，避免 sleep 依赖的 ticks 不增长而导致“时间显示卡住”。

- [workspace/OS/lab2/app/main.c](workspace/OS/lab2/app/main.c)
  - TEST_CASE_NONE 使用持续循环显示 RTC（便于演示每秒刷新）。
  - TEST_CASE_OOB/TEST_CASE_PRIV 保持 10 秒演示后触发异常，便于分开测试与截图。

### 4. 用户态异常测试

## 五、问题回答

### 问题 1

为什么用户程序不能直接调用内核函数，而要通过 int 0x80 这种特殊方式？

答：

用户程序运行在 Ring 3，内核运行在 Ring 0。直接调用内核函数会绕过硬件特权检查，破坏保护边界。系统调用通过 int 0x80 进入 IDT 中允许的门，由 CPU 完成受控的特权级切换，确保只有内核暴露的受管接口可被访问，从而保证安全性与隔离性。

### 问题 2

当用户程序执行 int 0x80 时，CPU 自动将哪些内容压栈？iret 返回时栈帧如何变化？

答：

发生特权级切换（Ring 3 -> Ring 0）时，CPU 会自动压入用户态返回现场：SS、ESP、EFLAGS、CS、EIP。随后中断入口代码再保存通用寄存器和段寄存器，组成完整 TrapFrame。

内核处理完后执行 iret，按相反顺序弹栈恢复：EIP、CS、EFLAGS，并在跨特权级返回时继续恢复 ESP、SS，回到用户态原执行点继续运行。

### 问题 3

在 syscallWrite 中，为什么要通过 getSegBase(tf->ds) 做地址转换？

答：

用户态程序使用的是自身段内虚拟偏移地址，内核不能把该偏移当作内核线性地址直接解引用。必须先取出用户数据段基址，再执行 物理可访问地址 = 段基址 + 用户偏移，才能正确访问用户缓冲区，避免读写错误区域或破坏内核内存。

### 问题 4

app 被 kernel 加载到内存后，为什么不能直接跳转执行？框架如何处理？

答：

因为 app 需要在 Ring 3 执行，且应使用用户段（基址映射到 APP_BASE）与用户栈；直接普通跳转仍停留在 Ring 0 和内核上下文中，不满足隔离与权限要求。

框架通过 enterUserSpace 调用 switch_to_user 构造 iret 返回帧（SS, ESP, EFLAGS, CS, EIP）并执行 iret，让 CPU 以硬件机制切换到用户态段与用户栈，再从用户入口地址开始执行。

## 六、实验结论

本实验完成了从用户态库到内核系统调用处理的完整链路，掌握了以下关键点：

1. IDT 门权限（DPL）对系统调用可达性的控制。
2. int/iret 触发与返回过程中的硬件栈帧机制。
3. 分段模型下用户地址到可访问地址的转换。
4. 定时器中断驱动的时间片计数与 sleep 基础实现。
5. 异常触发与内核保护机制验证。
