## 为什么需要系统调用
- 用户程序运行在**Ring 3（低特权级）**，内核运行在**Ring 0（高特权级）**
- Ring 3 代码不能：
  1. 直接访问硬件设备
  2. 执行特权指令（如 hlt、sti）
  3. 访问内核内存区域
- 操作系统提供**系统调用**作为安全接口

## 系统调用的设计思想
1. 用户程序调用库函数（如 printf）
2. 库函数通过软中断 `int 0x80` 触发特权级切换
3. CPU 自动从 Ring 3 → Ring 0
4. 内核执行对应服务，通过 `iret` 返回用户态

执行链路：
`printf("..")` → `app/main.c` → `lib/sysc...`（用户态库 Ring 3）→ `syscall()` → `int 0x80` → `kernel/x86/trap_entry` → `kernel/core/irq_dispatch`（中断分发）→ 执行具体服务（写显存/读RTC）（内核态 Ring 0）

## 软中断 int 指令
- `int n` 指令：触发中断号 n
- 常见用法：
  1. `int 0x80`：系统调用（Linux/本实验）
  2. `int 0x13`：BIOS磁盘服务（实模式）
  3. `int 3`：软件断点（调试）
- 执行 `int 0x80` 时：
  1. CPU 检查中断描述符表（IDT）
  2. 找到中断门，获取内核入口地址
  3. 切换到 Ring 0
  4. 跳转到内核中断处理程序

## 中断描述符表（IDT）
- IDT 记录了每个中断号对应的处理程序入口
- 本实验重要门：
  1. `0x80`：系统调用门（DPL=3，允许用户调用）
  2. `0x20`：时钟中断门（DPL=0，仅内核）
- 函数：`setIntr` 设置中断门，`setTrap` 设置陷阱门
- 中断门描述符结构：
  - offset：处理程序地址
  - segment：代码段选择子
  - DPL：特权级（3=用户可调）
  - type：STS_IG32（中断门）
- 类比：IDT 类似一本书的“目录”，中断号=页码，处理程序=实际内容

## 两种门的区别
| 特性 | 中断门 (STS_IG32) | 陷阱门 (STS_TG32) |
| ---- | ---- | ---- |
| IF 位 | 执行时清除（禁用中断） | 保持不变 |
| 适用场景 | 硬件中断 | 异常、系统调用 |
| 本实验用途 | 0x20 时钟、0x80 syscall | 其他异常 |

## 中断门描述符
关键位：
1. P = Present（是否存在）
2. DPL = Descriptor Privilege Level（允许调用的最低特权级）
   - DPL=0：仅Ring 0可调用
   - DPL=3：Ring 0和Ring 3都可调用

## DPL与CPL的关系
规则：**CPL ≤ DPL 才能调用**
1. 系统调用门（DPL=3）：
   - CPL=0（内核）：可以调用 ✓
   - CPL=3（用户）：可以调用 ✓
2. 仅内核可用的门（DPL=0）：
   - CPL=0（内核）：可以调用 ✓
   - CPL=3（用户）：不能调用 ✗（触发#GP）
- 本实验配置：
  - `int 0x80`：DPL=3 → 用户态可调用
  - `int 0x20`：DPL=0 → 仅内核可设置

## Int执行时CPU做了什么
- 栈变化方向：从高地址向低地址
- 执行 `int 0x80` 前：ESP → [用户数据...]
- CPU自动压栈：
  1. ESP → [EIP]（返回地址）
  2. [CS]（代码段）
  3. [EFLAGS]（标志寄存器）
  4. [ESP]（仅特权级改变时）
  5. [SS]（仅特权级改变时）
- 压栈后跳转到中断处理程序，`int` 指令由硬件自动完成，返回用 `iret` 恢复

## 中断入口代码保存寄存器的原因
中断入口代码：
```asm
asmDoIrq:
push ds
push es
push fs
push gs
pushad ; 保存8个通用寄存器（EAX, ECX, EDX, EBX, ESP, EBP, ESI, EDI）
; ... 处理中断 ...
popad
pop gs
pop fs
pop es
pop ds
iret
```
- 原因：中断处理完后要返回继续执行；被中断的代码不知道发生了中断；必须恢复所有寄存器状态

## iret：恢复现场，返回用户态
执行 `iret` 时：
1. 弹出 EIP → 恢复指令指针
2. 弹出 CS → 恢复代码段
3. 弹出 EFLAGS → 恢复标志位
4. 特权级改变时：
   - 弹出 ESP → 恢复到用户栈
   - 弹出 SS → 恢复到用户段
- `iret` 是唯一能从中断返回的指令
- 与 ret 区别：ret 只恢复 EIP

## 用户态地址转换
1. 转换流程：段选择子 → 查GDT → 得到段描述符 → 得到基址
2. 用户程序配置：
   - 代码链接地址：0x0
   - 数据链接地址：0x0
   - 实际加载物理地址：0x20000
3. 内核访问用户数据必须做地址转换，否则访问错误内存位置
4. 转换公式：**物理地址 = 段基址 + 偏移**
   示例：用户VA 0x10 → 物理地址=0x20000+0x10=0x20010

## 物理内存的划分（简化）
1. 0x00000：BIOS/中断向量
2. 0x10000：内核代码和数据（限长1MB）、内核栈（向下增长）
3. 0x20000：用户程序（Ring3），VA0x0映射到这里（限长128KB）
4. 0x40000：用户栈（向下增长），VA0x20000映射到这里
5. 0xB8000：VGA显存

## 定时器配置（PIT）
### 核心基础
- PIT（Programmable Interval Timer，可编程间隔定时器）：x86核心硬件定时器，通过16位计数器周期性递减产生固定频率信号，提供周期性时钟中断，系统累计滴答数实现时间管理
- 输入频率：1193182Hz（约1.19MHz）
- 目标：每10ms产生一次中断，频率=100Hz
- 分频值计算：1193182 ÷ 100 = 11931.82 → 取11932（十六进制0x2E9C）
- 写入端口0x40：先写低字节0x9C，再写高字节0x2E
- 效果：定时器每10ms向PIC发送一次中断信号

### 核心组成与通道分工
| 通道编号 | 端口号 | 核心用途 |
| ---- | ---- | ---- |
| 计数器0 | 0x40 | 产生系统时钟信号（连接IRQ0，工作方式3） |
| 计数器1 | 0x41 | 控制DRAM定时刷新 |
| 计数器2 | 0x42 | 驱动扬声器发声（输出特定频率方波） |

### PIT控制寄存器（端口0x43）配置规则
| 位范围 | 字段名称 | 作用 |
| ---- | ---- | ---- |
| bit 7-6 | 计数器选择 | 00=计数器0，01=计数器1，10=计数器2，11=无效 |
| bit 5-4 | 数据读写方式 | 00=锁存数据；01=仅读写低8位；10=仅读写高8位；11=先低后高8位 |
| bit 3-1 | 工作模式 | 共6种模式 |
| bit 0 | 计数方式 | 0=二进制计数（最大65535）；1=BCD码计数（最大9999） |

### 控制字配置
- SC=00（通道0）
- RL=11（先低字节后高字节）
- M=010（模式2：速率发生器）

## 系统调用号
### 调用约定
- 用户调用：`syscall(系统调用号, 参数1, 参数2, ...)`
- 寄存器约定：
  - EAX = 系统调用号
  - ECX, EDX, EBX, ESI, EDI = 参数1-5

### 本实验系统调用号
- 0 → SYS_WRITE：写屏
- 2 → SYS_NOW：获取时间
- 3 → SYS_ZERO_TICKS：重置ticks
- 4 → SYS_GET_TIME_TICKS：获取ticks

## 系统调用完整流程
1. 用户程序：`printf("Hello")`
2. 库函数：`lib/sysc...` 调用 `syscall(SYS_WRITE, buffer, len, ...)`
3. 汇编入口：`doSyscall.asm` 执行 `mov eax, 0` + `int 0x80`
4. CPU硬件：切换Ring 0 → 查IDT[0x80] → 跳转到irqSyscall入口
5. 中断保存：`trap_entry.asm` 保存寄存器 → 调用irqHandle
6. 中断分发：`irq_dispatch.c` 分发到syscallWrite
7. 内核执行：写VGA显存0xB8000
8. 返回用户态：`iret`

## General Protection Fault（#GP，通用保护异常）
### 触发条件
1. 访问超出限长的内存
2. 执行特权指令（Ring 3执行hlt/sti）
3. 访问DPL不满足的门
4. 读写不存在的中断门

### 错误代码格式
- bits 0-2：描述符类型
- bit 0：External（外部触发）
- bit 1：GDT/LDT（选择子类型）
- bit 2：IDT（是否为IDT项）

## 异常处理流程（本实验）
1. CPU检测到异常
2. 查IDT找到处理程序
3. 压栈保存现场
4. 跳转到异常处理程序
5. 处理程序分析错误原因
6. 输出调试信息
7. 停机或终止进程

### GProtectFaultHandle 功能
- 打印错误信息（EIP, CS, CPL等）
- 打印Halting system
- 进入死循环
