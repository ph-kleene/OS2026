# schedule 进程调度实现说明 (Lab 3)

本文档详细描述了项目中 `schedule` 函数的实现机制、具体的调度触发时机以及调度完成后底层进程切换的细节。

## 1. 调度算法与实现

### 调度逻辑 (`kernel/core/proc.c`)
项目采用 **逆向扫描 (Reverse Scanning)** 的简单公平调度算法：
1. **遍历 PCB**：从 `MAX_PCB - 1` 向 `0` 逆向遍历进程控制块数组。
2. **状态判定**：寻找第一个处于 `RUNNABLE`（就绪）或 `RUNNING`（当前运行）状态的进程。
3. **优先级倾向**：由于扫描顺序从高到低，具有较高 PID 的进程（通常是新创建的子进程）在同等状态下会优先获得执行机会。
4. **状态转换**：
   - 若选中的进程 `next` 不是当前进程 `current`：
     - 若 `current` 仍处于 `RUNNING` 状态（受抢占触发），则将其降级为 `RUNNABLE`。
     - 将 `next` 状态设为 `RUNNING`，并更新 `current` 指针。

## 2. 调度触发时机

系统在以下关键节点触发调度逻辑：

| 触发位置 | 触发原因 | 行为描述 |
| :--- | :--- | :--- |
| **`timerHandle`** | **进程唤醒** (休眠到期) | 时钟中断每 10ms 检查一次阻塞队列。当有进程休眠结束变为 `RUNNABLE` 时，触发调度以寻找当前最高优先级的就绪进程。 |
| **`do_sleep`** | **主动阻塞** (进入休眠) | 运行进程因调用 `sleep` 变为 `BLOCKED` 状态，必须立即触发调度释放 CPU 资源。 |
| **`do_exit`** | **进程终结** (任务结束) | 进程状态被置为 `UNUSED`，内核强制触发调度以切换至其他就绪进程。 |
| **`do_fork` / `do_vfork`** | **新任务就绪** (克隆完成) | 子进程创建并初始化为 `RUNNABLE`。为了让新进程尽早执行，内核在系统调用返回前主动触发一次调度。 |

## 3. 调度完成后的进程切换细节

当 `schedule()` 完成 `current` 指针的更新后，真正的“切换”是由内核在从中断返回（`iret`）的过程中配合硬件完成的：

### A. 内核栈与硬件上下文 (`trap_entry.asm`)
1. **保存现场**：在中断发生时，`asmDoIrq` 将 `esp` 保存到旧进程的 `pcb->tf`。
2. **切换栈指针**：中断处理结束返回汇编时，执行 `mov esp, [current]`。由于 `current` 已被修改，此时的 `esp` 指向了**新进程**的内核栈。
3. **恢复现场**：后续的 `popad` 和 `pop ds` 指令会从新进程的栈中恢复它上次被切走时的寄存器数值。

### B. 内存空间物理隔离 (`gdt.c`)
- **段基址切换**：内核调用 `update_user_seg_base`。
- **细节**：通过修改 GDT 中用户段（`SEG_UCODE/SEG_UDATA`）的基地址（`base`），将用户态的虚拟地址空间映射到该进程专属的 128KB 物理槽位。这样，即便所有用户程序都编译在 `0x0` 地址，硬件层面上也会访问不同的物理内存。

### C. 特权级转换准备 (`TSS`)
- **`tss.esp0` 更新**：内核更新任务状态段中的 `esp0`。
- **细节**：这确保了当新进程下一次因中断进入内核态时，CPU 能自动切换到它私有的 `kstack` 顶端，保证内核态执行环境的独立性。

### D. 最终跳转 (`iret`)
- 执行 `iret` 指令，CPU 弹出新进程的 `EIP`、`CS` 和 `EFLAGS`。至此，处理器正式跳转到新程序的指令流中执行。
