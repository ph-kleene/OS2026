# OS Lab3 实验报告

## 1. 实验信息
- **实验名称**：Lab3 进程管理与调度
- **实验目录**：`workspace/OS/lab3`
- **运行环境**：Linux + QEMU (`qemu-system-i386`)

## 2. 实验目的
- 在 Lab2 基础上，将系统从单进程环境扩展为支持多进程并发管理与抢占式调度的原型。
- 完成 `fork`、`vfork`、`exec`、`exit`、`sleep` 等系统调用以及调度器、GDT 动态重定位的实现。
- 验证进程控制的五项核心机制：fork 槽位上限、fork 内存独立、vfork 二次创建限制、vfork 内存共享以及基于 sleep 的进程调度切换。

## 3. 核心设计与实现说明

### 3.1 PCB 初始化与用户态切换
- **修改文件**：`kernel/kernel.c`
- **实现细节**：
  - 初始化系统进程控制块 `pcb[i]`，设置 `pid`、`ppid`、`state`、`mem_base`、`mem_limit` 及 `vfork_count` 的初始状态。
  - 构造 Idle 进程（PID 0）的 TrapFrame，通过 `iret` 指令完成从内核态到用户态的首次跃迁。

### 3.2 调度器与进程生命周期管理
- **修改文件**：`kernel/core/proc.c`
- **实现细节**：
  - **`schedule()`**：执行调度前先判断当前进程状态，再考虑是否将其状态修改为 `RUNNABLE`。采用逆向扫描（高 PID 优先）策略选择就绪进程，随后依次更新 `current` 指针、切换 TSS 内核栈顶，并调用 `update_user_seg_base` 切换用户段基址实现上下文切换。
  - **`do_fork()`**：扫描获取空闲 PCB 槽位，完整复制父进程的 128KB 物理内存、内核栈与 TrapFrame，并按约定设置父子进程的不同返回值。
  - **`do_vfork()`**：子进程复用父进程的 `mem_base` 实现共享内存。通过校验和递增父进程的 `vfork_count`，限制同一父进程并发未 exec 的 vfork 子进程数量。为确保专项测试现象可见，测试时临时增加了将父进程置为 `BLOCKED` 以及手动复制用户栈内容的逻辑。
  - **`do_exec()`**：加载磁盘程序。若调用者为 vfork 创建的子进程，则在此处为其分配独立的物理槽位解除内存共享，并递减父进程的 `vfork_count` 释放资源。
  - **`do_exit()`**：结束当前进程，将其状态置为 `UNUSED`。增加清理逻辑：递减父进程的 `vfork_count` 并恢复父进程用户栈，最后主动触发 `schedule()`。
  - **`do_sleep()`**：将进程状态置为 `BLOCKED` 并设定 `sleep_ticks`，主动放弃 CPU（非忙等实现）。

### 3.3 中断处理与段基址重定位
- **修改文件**：`kernel/core/irq_dispatch.c` / `kernel/x86/gdt.c`
- **实现细节**：
  - **定时器中断**：`timerHandle()` 在每次时钟中断（100Hz）时递减所有阻塞进程的 `sleep_ticks`，归零后将其状态唤醒为 `RUNNABLE`，并触发抢占式调度。
  - **系统调用分发**：补充了对 fork / vfork / exec / exit / sleep / getpid / getppid 等系统调用的中断分发逻辑。
  - **GDT 动态切换**：在 `update_user_seg_base(base, limit)` 中动态更新全局描述符表 `SEG_UCODE` 和 `SEG_UDATA` 的基址字段，实现基于物理槽位的内存隔离。

### 4.2 fork 内存独立性专项测试
**测试条件**：设置 `APP_TEST_MODE` 为 `APP_TEST_FORK_SHARE`。

![fork内存独立](lab3_04_fork_memory_isolation.png)

### 4.3 vfork 内存共享专项测试
**测试条件**：设置 `APP_TEST_MODE` 为 `APP_TEST_VFORK_SHARE`，并开启 `VFORK_MEMORY_TEST_MODE 1`。

*现象分析*：子进程通过 vfork 创建后，修改了全局变量 `g_shared` 为 300。父进程随后读取该变量，输出值为 300。这证明 vfork 创建的子进程在执行 exec 前，与父进程严格共享同一段物理内存空间。

## 5. 课后问答题

### 5.1 fork 与 vfork 的本质区别是什么？在内存管理上有何不同？
- **本质区别**：`fork` 用于创建一个与父进程完全独立的子进程副本；而 `vfork` 是一个轻量级的创建机制，主要设计意图是作为 `exec` 的前置操作，避免不必要的拷贝开销。
- **内存管理差异**：`fork` 被调用时，内核会为子进程分配全新的物理内存槽位，并完整复制父进程的地址空间（数据、代码、栈等），父子进程内存相互独立。`vfork` 被调用时，内核不复制内存，子进程与父进程共享同一物理地址空间（共用 `mem_base`），仅通过栈指针偏移来规避浅层的栈冲突。

### 5.2 vfork 为何需要限制其创建的子进程数量？它如何通过与 exec 的配合来控制子进程数量？
- **限制原因**：由于 `vfork` 的父子进程完全共享数据段和代码段，如果允许父进程同时拥有多个活跃的未执行 `exec` 的 vfork 子进程，极易引发严重的数据竞争和用户栈踩踏，破坏程序状态。
- **控制配合机制**：内核在 PCB 中维护了 `vfork_count` 字段。每次父进程调用 `vfork` 时，若 `vfork_count > 0` 则予以拒绝，否则将其递增。当 vfork 子进程调用 `exec` 加载新程序时，内核会解除内存共享状态，为该子进程分配独立的物理槽位，并**同步递减父进程的 `vfork_count`**，从而允许父进程在未来发起新的 `vfork` 调用。

### 5.3 描述 `schedule()` 中上下文切换过程。
在 `schedule()` 函数中，上下文切换经历了以下完整过程：
1. **状态保存与判断**：时钟中断或主动调用已将当前进程的寄存器现场保存至内核栈。调度器首先判断当前进程的状态，再决定是否将其从 `RUNNING` 置为 `RUNNABLE`。
2. **选择就绪进程**：采用逆向扫描算法（从高 PID 向低 PID 遍历 PCB 数组），挑选出一个状态为 `RUNNABLE` 或 `RUNNING` 的进程作为下一个执行对象。
3. **更新进程状态**：将选出的新进程状态从 `RUNNABLE` 更新为 `RUNNING`，并更新全局指针 `current` 指向该新进程。
4. **切换内核栈与 TSS**：更新当前 CPU 的 TSS（任务状态段）结构，将其 `esp0` 字段指向新进程的私有内核栈栈顶，以备下一次陷入内核时使用。
5. **切换用户段基址（GDT）**：调用 `update_user_seg_base()` 函数，根据新进程所在的物理槽位更新 GDT 中的用户代码段（`SEG_UCODE`）和用户数据段（`SEG_UDATA`）基址，完成物理内存隔离的切换。
6. **现场恢复**：函数返回，随后的汇编代码从 `current->tf` 指向的地址恢复新进程的通用寄存器，最后通过 `iret` 指令跳转至新进程的用户态断点继续执行。
