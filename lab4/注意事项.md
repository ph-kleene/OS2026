## 进程状态扩展
* **新增状态定义**
  * `UNUSED = 0` // 空闲
  * `RUNNABLE = 1` // 就绪
  * `RUNNING = 2` // 运行
  * `BLOCKED = 3` // 阻塞
  * `ZOMBIE = 4` // 新增：僵尸
  * `WAIT_CHILD = 5` // 新增：等待子进程
* **PCB新增字段**
  * `time_count`: 时间片计数器 (5 ticks = 50ms)
  * `next`: 信号量等待队列链表指针
  * `ppid`: 父进程ID (用于孤儿处理)
* **信号量结构 (`struct Semaphore`)**
  * `int value;` // 信号量值
  * `int used;` // 使用标志
  * `struct PCB *wait_queue;` // FIFO等待队列

## 信号量基础原理
* **什么是信号量？**
  * PV操作：进程间同步的经典方法
  * P操作 (wait)：申请资源，资源不足则阻塞
  * V操作 (post)：释放资源，唤醒等待进程
* **核心特性**
  * FIFO公平性：等待队列按顺序唤醒
  * 原子性：操作期间不会被中断打断
  * 资源计数：表示可用资源数量

## 需要大家实现的信号量操作
* **`sem_init` 信号量初始化**
  * 查找空闲信号量
  * 设置 `used = 1`, `value`, `wait_queue = NULL`，返回索引
* **`sem_wait` P操作**
  * **情况1：** `value > 0`，执行 `value--`，立即返回
  * **情况2：** `value == 0`
    * ① 加入 wait_queue (FIFO)
    * ② `state = BLOCKED`
    * ③ `eip -= 2` (Mesa重试)
    * ④ `schedule()` 让出CPU
    * ⑤ 返回 `SYS_SEM_WAIT`
* **`sem_destroy` 信号量销毁**
  * 检查有效性
  * 设置 `used = 0`, `wait_queue = NULL`，返回结果
* **`sem_post` V操作**
  * ① `value++`
  * ② 若 wait_queue 非空：唤醒队首进程，`state = RUNNABLE`，`next = NULL` (脱离队列)

## 时间片轮转算法
* **什么是时间片轮转 (Round-Robin)？**
  * 每个进程获得固定的CPU时间片
  * 时间片耗尽时强制切换到下一进程
  * 确保所有进程公平获得CPU时间
* **时间片参数**
  * 时间片长度：5 ticks = 50ms (每10ms递减)
  * 调度触发：`time_count <= 0`
* **Lab3与Lab4调度算法对比**

| 算法特性 | Lab3 | Lab4 |
| :--- | :--- | :--- |
| **扫描方向** | 逆向扫描 | 轮转扫描 |
| **优先级策略** | 高PID优先 | 公平轮转 |
| **是否抢占** | 无 | 抢占式 |

## Schedule函数实现
* ① 扫描起点 = `(current->pid + 1) % MAX_PCB`
* ② 循环查找 RUNNABLE/RUNNING 进程
* ③ 优先调度用户进程 (PID > 0)
* ④ 无用户进程时调度 Idle (PID 0)
* ⑤ 上下文切换：`current = next`，`time_count = 5`，更新TSS和GDT

## timerHandle时钟中断
* ① sleep进程唤醒：递减 `sleep_ticks`，为0时设为 RUNNABLE
* ② 时间片抢占：`current->time_count--`，`<= 0` 时调用 `schedule()`
* ③ 调度触发：无论当前进程是否 RUNNING，都调用 `schedule()`

## wait/exit系统调用
* **进程状态转换**
  * `RUNNING -exit-> ZOMBIE -wait-> UNUSED`
  * 子进程退出，父进程回收
* **`do_exit`**
  * **情况1：** 父进程正在等待 (`state == WAIT_CHILD`)。将当前进程设为 ZOMBIE，等待父进程调用 wait 回收。
  * **情况2：** 父进程已经不存在或不在等待。将当前进程设为 UNUSED，自己释放 PCB 槽位。
* **`do_wait`**
  * 检查子进程存在性：有活跃子进程？阻塞等待；全为僵尸？清理回收。

## 温馨提醒（实验核心注意事项）
* 本次实验框架在修改前存在以下问题：如果父进程没有调用 `wait` (即父进程没有处于 `WAIT_CHILD` 状态)，退出进程会永久保持 ZOMBIE 状态，不会变为 UNUSED。
* 这是一种僵尸进程泄露的情况。
* 此时父进程需要主动调用 `wait` 系统调用来回收子进程的资源，将其状态从 ZOMBIE 改为 UNUSED。
* 因此，我们更正了框架代码并增加了需要大家在 `main.c` 中自行测试的 TODO，即在 `do_exit` 中需要考虑两种情况：考虑父进程 `wait` 或者没有 `wait` 两种情况。
* 如果你正确实现了这部分，即使父进程没有调用 `wait`，子进程 `exit` 后是会被正确回收的 (槽位被释放)。
* TODO中的测试方案可能不是最优的，我们欢迎大家提出更好的测试方案并写在实验报告中。
